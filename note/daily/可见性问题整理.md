# volatile 与 final 的可见性与指令重排

## 一、整体知识点整理

### volatile 的内存语义

**可见性保证**
- 写入 volatile 变量时，强制将值从 CPU 缓存刷回主内存
- 读取 volatile 变量时，强制从主内存读取而非 CPU 缓存
- 保证所有线程始终看到 volatile 变量的最新写入值

**指令重排保证**
- volatile 写之前的所有操作，不会被重排到 volatile 写之后
- volatile 读之后的所有操作，不会被重排到 volatile 读之前
- volatile 写插入 `StoreStore +volatile写+ StoreLoad` 内存屏障，volatile 读插入 `volatile读 + LoadLoad + LoadStore` 屏障

**连带可见效果**：volatile 写之前的所有普通写（如写A），会被 StoreStore 屏障强制刷到主内存。其他线程在读 volatile 时，不仅看到 volatile 的最新值，还能连带看到写之前所有普通变量的最新值。

### final 的内存语义

**可见性保证**
- 构造器完成后，任何线程看到该对象引用时，保证能看到 final 字段的正确初始化值
- 这是对象安全发布的核心机制，不需要额外同步手段

**指令重排保证**
- 构造器内对 final 字段的写入，不会被重排到构造器外的引用赋值之后
- 编译器在构造器 return 前插入 `StoreStore` 屏障

**屏障特点**：屏障插在构造器末尾，保护整个构造器内所有 final 字段的写入。多个 final 字段共享同一道屏障，不是每个字段单独有一道屏障。

### volatile 与 final 的核心区别

| 维度 | volatile | final |
|---|---|---|
| **可见性** | 每次读/写都跨线程可见 | 构造完成后可见（一次性） |
| **重排** | 读写两侧都加屏障 | 仅构造器内加屏障，防止逸出 |
| **语义** | 可变数据的线程安全读取 | 不可变数据的安全发布 |
| **屏障位置** | 每次读写该字段时插入 | 构造器末尾（只有一处） |
| **触发时机** | 每次访问都触发 | 构造完成那一刻触发一次 |
| **成本** | 较高（每次读写都刷） | 极低（只在构造器末尾刷一次） |

### 内存屏障类型

四种屏障的本质是一道"闸门"，闸门左边的操作必须全部落定后，右边的操作才能开始。

| 屏障 | 含义 | 一句话理解 | 实际行为 |
|---|---|---|---|
| **StoreStore** | 写→写 | 前面的写必须先生效，后面的写才能执行 | 先刷写缓冲到主内存，再允许后续写操作进入写缓冲 |
| **StoreLoad** | 写→读 | 前面的写必须先生效，后面的读才能执行（最强屏障） | 先刷写缓冲到主内存，再从主内存读取数据 |
| **LoadLoad** | 读→读 | 前面的读必须先完成，后面的读才能执行 | 先完成前一次读（可能涉及缓存失效），再允许后续读操作 |
| **LoadStore** | 读→写 | 前面的读必须先完成，后面的写才能执行 | 先完成读操作，再允许后续写操作进入写缓冲 |

Tip: Store在前面就是将写缓存刷入内存；Store在后面就是写入写缓冲

**不同架构的内存模型差异**

| 架构 | 内存模型 | 允许的重排 | 屏障开销                |
|---|---|---|---------------------|
| **x86** | 强模型 | 仅 Store-Load | StoreLoad允许重排，其他不允许 |
| **ARM** | 弱模型 | 四种全允许 | 四种屏障都允许             |

**x86 特点**：只允许写→读重排，其他三种天然禁止。volatile 写只需一条 `mfence`（StoreLoad），编程简单但内存操作成本较高。

**ARM 特点**：四种重排都可能发生，需要完整屏障序列。volatile 写需要真实插入 StoreStore + StoreLoad，volatile 读需要 LoadLoad + LoadStore。编程复杂但内存操作本身更灵活高效。

**StoreLoad 为最强屏障的原因**：CPU 写操作是异步的（数据先进缓冲区，稍后才真正写到内存），而读操作要等内存返回。StoreLoad 要求写必须真正落盘之后才能读，相当于把写缓冲全部刷空，代价最高。

### 内存屏障的双重作用

同一道屏障同时解决重排和可见性两个问题，不是两组不同的机制：

| 效果 | 实现方式 |
|---|---|
| **防重排** | 闸门不许操作跨过（解决顺序问题） |
| **保可见** | 闸门开时强制刷缓存/失效缓存（解决缓存问题） |

### final 可见性的完整链路

```
①② 写final字段
    ↓ [StoreStore屏障] → 强制刷写缓冲到主内存(构造方法的结尾)
④ 写引用
    ↓ 另一个线程看到引用 ≠ null
⑤ 读引用
    ↓ [LoadLoad屏障] → 强制失效本地缓存
⑥ 读final字段 → 从主内存读，必然看到①②的值
```

关键推理：如果引用已可见（≠ null），说明④已生效，屏障保证了①②在④之前生效，所以①②也必然可见。

### 重要陷阱：构造器内引用逸出

构造器内不能让 `this` 逸出，否则屏障还没生效，其他线程就已经通过逸出的引用去读 final 字段，可能读到未初始化的值。

## 二、疑问解答

### Q1：给类每个属性加上 final 的意义是什么？

不是让属性逐个排队写，而是让 **每个属性都被屏障保护**。StoreStore屏障只插在构造器末尾，所有 final 字段的写入都不会越过屏障逃逸到构造器外面。对于非 final 字段，它的写入可能被重排到引用赋值之后，其他线程可能读到未初始化的值。

一句话总结：final 的保证不是"写完一个再写下一个"，而是 **"构造器内的所有 final 写入，一定在对象引用被外界看到之前全部生效"**。

### Q2：volatile 的屏障与 final 的屏障本质区别？

理解方向正确。volatile 屏障绑在字段上，每次读写该字段时插入多道屏障，该字段本身是"屏障的锚点"。final 屏障绑在构造器上，屏障插在构造器 return 之前，所有 final 字段被动地被这道屏障保护，final 字段本身不会触发任何屏障。

比喻：volatile 是每个字段自带"安检门"，每次进出都要过安检；final 是构造器出口设了一道"大门"，所有 final 字段在门内写完，门开了才能出去。

### Q3：volatile 写之前的 StoreStore 屏障会把写A也刷到内存吗？

是的，写A会被 StoreStore 屏障强制刷到主内存。这正是 volatile 写的核心语义："我写之前的一切，都要先落盘"。StoreStore 屏障触发时，会把写缓冲里所有内容（包括写A）全部刷到主内存。

连带可见效果：其他线程在读 volatile 时，不仅看到 volatile 的最新值，还能连带看到写之前所有普通变量的最新值。这是 StoreStore 屏障在写侧、LoadLoad 屏障在读侧配合的结果。
